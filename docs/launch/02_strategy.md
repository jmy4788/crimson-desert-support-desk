# 02 Strategy

Updated: 2026-03-31 (Asia/Seoul)

## Executive Summary

**Inference / hypothesis**

This project should not position itself as a general gaming guide site.

Its moat is not the codebase.
Its moat is:

- official-source traceability
- Korean editorial quality
- issue update speed
- symptom-first organization
- explicit trust labels on guidance

The product should win by being the most trustworthy unofficial Korean support hub for Crimson Desert, not by publishing the most content.

## Launch Goal

**Inference / hypothesis**

The immediate goal is not "perfect product."
The immediate goal is:

- launch a credible unofficial public version
- get indexed
- earn initial search traffic
- learn which issue pages actually pull demand

## 30-Day Launch Cutoff

### Must Have Before Public Launch

**Inference / hypothesis**

- public deployment
- real domain + HTTPS
- visible unofficial disclaimer in header/footer and key detail pages
- privacy page
- disclaimer page
- affiliate disclosure placeholder
- sitemap
- robots.txt
- analytics
- Google Search Console / Bing Webmaster submission
- manual review pass on top issue pages
- core issue pages with trustworthy structure

### Must Improve Before Public Launch

**Inference / hypothesis**

- cleanup imported issue slugs/titles
- add clearer status wording
- expose official source and latest checked time more clearly
- add stronger internal links between issue / patch / FAQ
- standardize trust labels:
  - official guidance
  - official-guidance-based interpretation
  - community reference

### Explicitly Cut for First Launch

**Inference / hypothesis**

These should not block launch:

- admin moderation dashboard
- full editorial queue UI
- advanced internal ops tools
- broad community automation
- premium membership
- paid app features
- multi-game expansion
- broad gaming editorial/news expansion

## Operating Policy

**Inference / hypothesis**

Public policy should be explicit, not implied.

Recommended operating rules:

- state on all major pages that this is not an official Pearl Abyss support property
- minimize official logo/art reuse
- show source link and latest checked time on every important page
- label every recommendation by trust level
- do not promise that a fix is guaranteed
- avoid unsafe registry / file-edit / unverified community fixes by default
- restrict affiliate links to clearly relevant pages only
- do not let monetization blocks crowd troubleshooting steps

## Naming / Brand Risk

**Inference / hypothesis**

`Support Desk` can sound official.

Public brand copy should likely be softer and clearer, for example:

- unofficial Crimson Desert support hub
- fan-made Crimson Desert issue tracker
- unofficial Crimson Desert help hub

Internal codename can stay as-is.
Public-facing naming should be reviewed before launch.

## Revenue Strategy

### Separate the Criteria

**Inference / hypothesis**

Do not rank monetization on one axis only.

Use three separate questions:

1. What is easiest to implement?
2. What can produce first revenue fastest?
3. What has the best long-term upside?

### Current Internal View

**Inference / hypothesis**

Easiest to implement:

- display ads

Potentially fastest to first revenue:

- affiliate links tied to relevant support intent

Better later-stage upside:

- newsletter sponsorship
- direct sponsor placements

Deprioritized for first phase:

- premium tools
- subscriptions
- donations as primary model
- broad media pivot

## Product Moat Statement

**Inference / hypothesis**

If we have to summarize the moat in one paragraph:

This product wins if users trust it more than generic guide sites.
That trust comes from official-source traceability, Korean editing quality, fast issue updates, and disciplined trust labeling.
If those weaken, the project becomes replaceable.
