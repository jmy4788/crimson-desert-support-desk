# Crimson Desert Support Desk 실제 프로토타입 서비스화 메모

기준일: 2026-03-30

---

## 1. 한 줄 결론

현재 상태에서 **로컬 데모를 넘어서 “실제로 써볼 수 있는 프로토타입 서비스”**까지 가는 데 필요한 시간은 대략 이 정도로 보는 게 맞다.

- **가장 빠른 내부용 프로토타입**: `3~5일`
- **실제 외부 공개 가능한 프로토타입**: `7~10영업일`
- **운영 루틴까지 포함한 공개 베타 수준**: `10~15영업일`

전제:

- 이미 기본 앱 구조와 샘플 데이터 구조는 있다.
- 지금 부족한 것은 “화면 틀”이 아니라 **실제 운영 데이터 흐름**이다.
- 즉, 남은 일의 핵심은 프론트 새로 짜는 게 아니라 **콘텐츠 ingestion + 검수 + 배포 + 운영 루틴**이다.

---

## 2. 지금 왜 내용이 비어 보일 수 있나

현재 코드 기준으로는 홈, Patch Hub, Known Issues, FAQ가 **샘플 데이터 기반으로 채워지도록** 되어 있다.

관련 위치:

- `data/seeds/sample_data.json`
- `backend/app/main.py`
- `frontend/src/lib/api.ts`

즉, 화면 뼈대만 있고 내용이 없는 구조는 아니다. 비어 보였다면 보통 아래 셋 중 하나다.

### A. 백엔드가 안 떠 있음

프론트는 기본적으로 `http://127.0.0.1:8000`의 API를 본다.

즉:

- 프론트만 켜면 UI는 보이지만
- API 응답이 없으면 실제 리스트/상세는 비어 있거나 에러 상태가 된다

### B. DB가 시드되지 않았거나 초기화가 꼬였음

샘플 데이터는 `sample_data.json`에 있지만, 실제 UI는 SQLite DB를 읽는다.

현재 로컬 기준으로는 이미 아래처럼 시드가 되는 상태다.

- sources: 6
- patches: 3
- issues: 8
- faq_entries: 10
- workaround_steps: 16
- landing_pages: 6

즉 DB만 정상 연결되면 최소 데모 데이터는 보여야 한다.

### C. “보이는 내용”은 있지만 아직 실제 운영 데이터가 아님

이게 더 중요한 포인트다.

지금 있는 내용은:

- 샘플 패치 3개
- 샘플 이슈 8개
- 샘플 FAQ 10개

실제 서비스 프로토타입으로 보이려면 필요한 건:

- 공식 공지 기반의 실제 데이터
- 더 많은 상세 페이지
- 운영자가 실제로 업데이트할 수 있는 입력 경로
- 배포된 환경에서 안정적으로 보이는지 검증

즉 지금은 **UI 데모 + 구조 검증 상태**고, 아직 **운영형 프로토타입 상태는 아니다.**

## 3. 실제 프로토타입으로 가려면 뭘 더 만들어야 하나

핵심은 5개다.

### 3-1. 실제 데이터 입력 경로

지금은 샘플 JSON만 있다. 실제 운영하려면 최소한 아래 중 하나는 필요하다.

#### 가장 빠른 방식

- 운영자가 `JSON` 또는 `Markdown + frontmatter` 파일을 직접 작성
- `scripts/import_official_source.py` 같은 CLI로 DB 반영

장점:

- 가장 빨리 완성 가능
- 정확도 통제 쉬움
- 복잡한 관리자 페이지 없이도 운영 가능

단점:

- 사람이 직접 넣어야 함

#### 한 단계 더 나은 방식

- 로컬 관리자 페이지 추가
- “공식 URL / 제목 / 날짜 / 원문 / 요약 / 관련 이슈” 입력 폼 제공
- 저장 시 DB 반영 + 검색 인덱스 재생성

장점:

- 비개발자 운영 가능
- 작업 속도 빨라짐

단점:

- 1~2일 추가 소요

### 3-2. 실제 콘텐츠 템플릿 고도화

지금 상세 페이지는 기본적으로 잘 뜨게 되어 있지만, 실제 서비스처럼 보이려면 항목 밀도를 더 높여야 한다.

각 페이지에 최소 아래는 있어야 한다.

#### Patch Detail

- 3줄 요약
- 이전 버전 대비 diff
- 영향 플랫폼
- 관련 known issues
- 출처 링크
- 최신 확인 시각

#### Issue Detail

- 증상 요약
- 현재 상태
- 영향 플랫폼
- 안전한 workaround 순서
- 라벨: `공식 안내 / 공식 안내 기반 해석 / 추론`
- 관련 패치
- 신고 권장 여부
- 출처 링크

#### FAQ Detail or FAQ Hub block

- 질문
- 답변
- 관련 이슈
- 관련 패치
- 출처 링크

### 3-3. 운영용 데이터 검수 규칙

실제 서비스는 “페이지가 있느냐”보다 “틀린 말을 안 하느냐”가 더 중요하다.

운영 규칙은 최소 아래 수준으로 강제해야 한다.

- 공식 URL 없으면 게시 금지
- 게시일/확인일 없으면 게시 금지
- workaround step에는 반드시 라벨 부여
- `추론` 라벨은 기본 2단계 이하로만 노출
- 위험한 설정 변경, 레지스트리 수정, 출처 불명 커뮤니티 팁은 기본 비노출

### 3-4. 배포 구조

프로토타입 서비스 단계에서는 아래 구성이 제일 현실적이다.

#### 추천

- Frontend: Vercel 또는 Netlify
- Backend: Render / Railway / Fly.io 중 하나
- DB:
  - 아주 초기에는 SQLite 단일 인스턴스도 가능
  - 외부 공개 시작 시점부터는 Postgres 전환 권장

이유:

- SQLite는 로컬/단일 인스턴스 데모엔 좋다
- 하지만 여러 배포 환경, 재배포, 동시성, 백업까지 생각하면 운영형에선 불안하다

즉:

- **내부/비공개 프로토타입**: SQLite 유지 가능
- **공개 프로토타입**: Postgres 전환 추천

### 3-5. 최소 운영 루틴

이 서비스는 결국 “얼마나 빠르게 정리해 올리느냐”가 경쟁력이다.

최소 루틴:

1. 하루 1회 공식 공지 확인
2. 새 패치나 known issue 발견 시 30분 내 초안 작성
3. 기존 issue 상태를 주 2회 재검토
4. 해결된 항목은 `resolved` 또는 `partially_improved`로 갱신

## 4. 가장 현실적인 구현 순서

현재 코드베이스 기준으로는 아래 순서가 가장 효율적이다.

### Phase 1. 데모를 실제 데이터 연결 구조로 바꾸기

예상: `1~2일`

할 일:

- API health 상태를 프론트에 노출
- API 실패 시 “백엔드 미실행 / 데이터 없음” 메시지 명확화
- 시드 재실행 버튼 또는 CLI 문서화
- 리스트/상세 empty state를 운영 친화적으로 수정

결과:

- “왜 비어 있는지” 바로 알 수 있음
- 로컬 시연이 훨씬 안정적임

### Phase 2. 수동 ingestion 파이프라인 만들기

예상: `2~3일`

할 일:

- `raw_sources/` 폴더 추가
- patch / faq / known issue용 입력 템플릿 정의
- `import_official_source.py` 스크립트 작성
- 입력 파일 검증
- DB upsert
- 검색 인덱스 재생성

결과:

- 샘플 JSON이 아니라 “실제 공식 데이터”를 넣을 수 있음

### Phase 3. 운영 콘텐츠 1차 생산

예상: `2~4일`

최소 목표:

- 실제 패치 5~10개
- 실제 known issue 10~20개
- 실제 FAQ 15개 이상
- 대표 검색 랜딩페이지 10개 이상

이 단계가 끝나야 비로소 서비스처럼 보인다.

### Phase 4. 배포와 외부 점검

예상: `1~2일`

할 일:

- 프론트/백 배포
- 환경변수 분리
- CORS 점검
- production DB 연결
- smoke test

### Phase 5. 공개 전 정리

예상: `1~2일`

할 일:

- disclaimer 강화
- sitemap / robots / canonical 확인
- 404 / empty / loading 상태 정리
- 모바일 점검
- 출처 누락 페이지 점검

## 5. 일정표로 보면 이렇게 잡는 게 맞다

### 5일 버전: “내부용 실제 프로토타입”

Day 1

- API 상태/empty state 정리
- 시드/실행 동선 정리

Day 2

- 실제 데이터 입력 템플릿 정의
- import CLI 초안 작성

Day 3

- patch / known issue / faq 실제 데이터 1차 입력

Day 4

- 검색/상세 연결 검수
- 라벨/출처 검수

Day 5

- 로컬 시연 정리
- 운영 문서 정리

### 8~10일 버전: “공개 가능한 프로토타입”

Day 1~2

- 내부용 프로토타입 안정화

Day 3~5

- 실제 데이터 대량 입력
- 관리자용 최소 입력 경로 또는 import CLI 확정

Day 6~7

- 배포
- 환경 변수
- staging 검수

Day 8~10

- 모바일/SEO/출처/로딩 상태 정리
- 공개 점검

## 6. 가장 빠른 정답은 무엇인가

만약 “진짜 빨리 서비스처럼 보이게”가 목표라면, 아래가 정답이다.

### 추천 전략

1. 관리자 페이지는 잠시 미룬다
2. 먼저 `공식 데이터 수동 입력 CLI`를 만든다
3. 실제 패치/이슈/FAQ를 20~30개 채운다
4. 그 상태로 먼저 배포한다
5. 운영 피로가 확인되면 그때 관리자 페이지를 만든다

이유:

- 지금 병목은 UI가 아니라 데이터다
- 관리 UI를 먼저 만드는 건 보기엔 좋아도 출시를 늦춘다
- 지금 필요한 건 “빠르게 신뢰 가능한 실제 콘텐츠를 넣는 것”이다

## 7. 내가 보는 가장 현실적인 일정 추정

현재 저장소 상태 기준으로 냉정하게 보면:

- **내가 바로 이어서 작업하면**
  - 내부 데모를 실제 프로토타입 수준으로 끌어올리는 데: `약 4~6일`
  - 외부 공개 가능한 프로토타입까지: `약 7~10일`

단, 아래가 있으면 더 빨라진다.

- 공식 공지 원문 링크 목록이 이미 정리돼 있음
- 우선순위 이슈 10~20개가 미리 정해져 있음
- 배포 대상이 이미 정해져 있음

아래가 없으면 더 느려진다.

- 어떤 공식 페이지를 기준 소스로 쓸지 아직 안 정함
- 실제 입력할 패치/FAQ/이슈 목록이 아직 없음
- 운영자가 수동으로 입력할 형식이 아직 없음

가장 효율적인 다음 작업 3개는 이거다.

2. `import CLI` 구현

이 셋이 끝나면, 지금 보이는 “빈 틀” 문제는 거의 사라진다.

## 9. 결론

지금 프로젝트는 실패한 상태가 아니라 정확히 **“UI 데모는 끝났고 운영 데이터 레이어가 아직 없는 상태”**다.

따라서 질문의 핵심은 “이걸 다시 만들 수 있냐”가 아니라:

- 실제 데이터를 어떤 형식으로 넣고
- 누가 검수하고
- 얼마나 빨리 배포 가능한 형태로 묶을 수 있냐

이쪽이다.

가장 현실적인 목표는:

- **이번 주 안에 내부용 실제 프로토타입**
- **다음 주 안에 공개 가능한 프로토타입**

이다.

그리고 가장 먼저 손대야 하는 건 프론트 새 디자인이 아니라 **실데이터 입력 파이프라인**이다.
